# AI Contract Review · 智能合同审查工作台

**合同审查不应该止于"现在打开 Word 自己改吧"。**

## TL;DR

把 `.docx` 合同拖进来。平台在同一个工作台里替你做完五件事：

1. **风险识别** —— 标出高风险/中风险条款，说清楚原因，给出具体修改建议。
2. **一键修复** —— 任意（或全部）建议都能一键应用，文档原地更新。
3. **行内批注** —— AI 批注 + 手动批注共存，点击批注跳到对应位置。
4. **合同对话** —— 直接问文档问题（"违约责任上限是多少？"），AI 基于本合同回答，引用条款。
5. **Word 导出（保留批注）** —— 包括每条批注的"已修复 / 未修复"状态，对方律师能直接看到改了什么、为什么改。

不需要装 Word 插件。审阅方不需要 Word 授权。最终产出是一份**可以直接发给对方的 .docx**。

## 核心洞察

今天主流的 AI 合同审查工具（Spellbook、Diligen、LawGeex）几乎都是 **Microsoft Word 插件**。这个选择本身就回答了"为谁设计的"——给已经住在 Word 里、不打算换工具的资深律师。

但合同审查的真正瓶颈不在律师在哪里办公，而在**工作流**：

> **读风险 → 起草修改 → 应用修改 → 导出 → 交付。**
> 今天的工具只做了第 1 步，第 2–5 步还是要律师手动。

如果 AI 标出了 12 条风险，律师还得在 Word 里手改 12 段，AI 大概只省了 5 分钟。剩下 90% 的工作没变。

AI Contract Review 走相反的路：**AI 的工作不是"标出来"，是"做完"。** 平台拥有从"识别风险"到"导出 Word"的完整闭环，让律师拿到的是**已经能直接发出去的文件**。

## 真实使用场景

**1. 大批量供应商 MSA 审查。**
采购部门一个季度收 30 份 MSA，长得几乎一样。律师 80% 的时间花在同样的六类高风险条款上（责任、知产、终止、赔偿、管辖、付款）。平台抓全这 6 类、给红线建议、导出干净版本。

**2. 创业公司 NDA 分诊。**
创始人收到一份 NDA，没有常驻律师。平台标出非标准条款（互保 vs 单方、永久保密、第 7 条藏的 IP 转让），用人话解释，然后让创始人一键改回去。

**3. 律师介入前的 term sheet 红线。**
创始人先用平台把 term sheet 过一遍，搞清楚哪些条款该 push back，再约律师。律师的计费时间从"我先解释这是什么意思"开始变成"我们直接谈判吧"。

## 工作机制

```
上传 .docx
     │
     ▼
┌───────────────────────────────────────────────────────────┐
│  浏览器工作台                                              │
│  ┌─────────────┐  ┌─────────────────┐  ┌──────────────┐   │
│  │ 侧边栏      │  │ 编辑器 (canvas) │  │ AI 面板      │   │
│  │ • 风险列表  │◀─│ • 行内文本      │─▶│ • 对话       │   │
│  │ • 批注      │  │ • 批注          │  │ • 摘要       │   │
│  │ • 一键修复  │  │ • 编辑 / 撤销   │  │ • 严格度滑块 │   │
│  └─────────────┘  └─────────────────┘  └──────────────┘   │
└───────────────────────────────────────────────────────────┘
     │
     ▼
导出 → Python 后端（python-docx）保留所有批注、
       修复状态、和原文档格式
     │
     ▼
可直接交付的 .docx
```

前端：React 19 + TypeScript + canvas-editor（Word 风格所见即所得）。
后端：Flask + python-docx（保留 Word 格式 + 批注注入）。
LLM：定制 prompt 工程的风险识别，三档严格度。

## 关键产品决策

**1. 做工作台，不做 Word 插件。**
Word 插件把你绑死在桌面、绑死在某个 OS、绑死在 Word 自己的 UX 里。浏览器工作台跨设备、未来支持多人协作、UX 完全自己掌控。代价是需要自己重建一个 Word 风格的编辑器；收益是**拥有完整的工作流闭环**。

**2. "一键修复"才是产品的本体。**
"AI 标出风险"任何人都能做。难的是让 AI 的建议**可执行**——能在原位置应用、能保留格式、能撤销、能批量。没有一键修复，AI 就是一个更复杂的 Ctrl+F。有了它，律师的工作流缩短一个数量级。**这一个决策就是 demo 和产品的分水岭。**

**3. 三档严格度（严格 / 均衡 / 宽松）。**
来自用户调研。一个一季度审 30 份 MSA 的采购律师要"严格"模式；一个第一次读 NDA 的创始人觉得"严格"太吓人。一刀切是糟糕的 UX。这个滑块加上去 5 秒，但它彻底改变了非律师用户的可达性。

**4. 自研批注系统，不用编辑器原生的。**
canvas-editor 不支持原生 Word 批注。可以换编辑器（OnlyOffice 是候选——但很重，集成复杂）。我们的选择：保留 canvas-editor 的编辑 UX，**单独做一个 Python 批注服务**，通过文本匹配在导出的 `.docx` 里注入批注。代价是后端复杂度更高，收益是前端轻量、批注行为完全可控。

**5. 导出 Word，不导出 PDF。**
PDF 是死文档。律师会拒绝任何不返回 `.docx` 的工具，因为下一步**永远是**在 Word 里继续谈判。锁死输出格式为 `.docx`（且保留批注）是这个用户群唯一能接受的交付格式。

**6. 合同对话基于文档本身，不是泛聊。**
对话面板把当前合同的文本注入到每个 prompt 里。问"违约责任上限是多少？"得到的是**针对本合同**的答案，并引用条款编号。一个泛用聊天机器人对律师没价值；一个"基于本合同"的聊天机器人变成研究工具。

## 开始使用

```bash
# 前端
npm install
npm run dev          # http://localhost:5173

# 后端（另一个 terminal）
cd backend
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env  # 填入 LLM API key
python app.py        # http://localhost:8000
```

把 `.docx` 合同拖进上传页，选严格度，开始审。

## 同类工作对比

| | 它做什么 | AI Contract Review 的差异 |
|---|---|---|
| **Spellbook / LawGeex / Diligen** | Word 插件里的 AI 风险识别 | 锁在 Word + 桌面。我们在浏览器、且拥有完整的"编辑 → 导出"闭环。 |
| **Harvey AI** | 律师 LLM Copilot，泛对话 | 没有结构化审查工作流，没有原位编辑。我们是 workflow-first。 |
| **DocuSign Insight / Kira** | 企业级合同分析 | 给 AML / 并购尽调用，规模化分析。我们给单份合同**主动审查**用。 |
| **直接问 ChatGPT** | 自由问答 | 没有批注、没有编辑、没有导出、没有风险标记。超过一段就难用。 |
